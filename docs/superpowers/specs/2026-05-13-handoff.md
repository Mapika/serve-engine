### Sub-project A — three follow-ups from live verification

Live-verification of A surfaced three issues that landed as their own
commits:

1. **`14b4abe fix(lifecycle):`** `serve run` replaces prior deployment of
   same name. The CLI promised "Stops the current model first" but the
   manager only evicted under VRAM pressure — two same-name deployments
   could coexist on a roomy GPU, with the proxy's
   `find_ready_by_model_name LIMIT 1` masking the config drift.
   `manager.load` now stops every prior ready deployment with matching
   `model_id` before placement; pinned ones force a clear 409 with
   "run `serve unpin <X>` before replacing it" instead of a bare 500.

2. **`2a42684 feat(adapters):`** parse + validate LoRA rank before
   engine load. Migration 007 adds `adapters.lora_rank` (parsed from
   `adapter_config.json` at pull time) and `deployments.max_lora_rank`
   (mirrored from operator's `-x '--max-lora-rank=N'`).
   `ensure_adapter_loaded` now pre-flights the rank with a clear
   "restart the deployment with -x '--max-lora-rank=N' (or higher)"
   error instead of letting vLLM produce a cryptic 500 on first request.

3. **`dd7f6f7 fix(proxy):`** populate `usage_events` tokens (incl. SSE
   `[DONE]` bug). `record()` now returns the inserted id; the streamer's
   `finally` block patches `tokens_in/out` from the upstream `usage`
   block. Found a streaming bug along the way: `_UsageTracker` was
   overwriting the last event with `data: [DONE]`, clobbering the
   usage payload from the prior frame — now only events containing
   `"usage"` are retained.

### Sub-project B — complete and live-verified

Five commits land the snapshot system:

- `9a55531 feat(snapshots):` backend snapshot interface
  (`supports_snapshots`, `snapshot_mount`, `snapshot_load_argv`,
  `snapshot_env`) + vLLM/SGLang opt-in. GPUInfo gained `compute_cap`
  for the snapshot key.
- `8e173d6 feat(snapshots):` lifecycle integration. `manager.load`
  computes the content-addressable key, pre-creates
  `~/.serve/snapshots/<key>/`, merges bind-mount + env + argv, and
  either touches the existing row (HIT) or fires a background save
  task (MISS) at `snapshot_save_delay_s` (10s prod, 0 tests) with a
  retry at +30s on failure.
- `57d06d0 feat(snapshots):` `serve snapshot ls/rm/gc` CLI + admin
  endpoints. GC is the two-step policy (keep-N per (engine, hf_repo);
  optional global `--max-disk-gb` LRU cap).
- `e92b8f5 fix(snapshots):` use engine-specific cache-root env vars.
  Live verification revealed vLLM 0.20.2 **ignores
  `TORCHINDUCTOR_CACHE_DIR` for its own compile cache** — it writes
  under `VLLM_CACHE_ROOT/torch_compile_cache/`. Each backend now
  overrides `snapshot_env` to point at the right var (vLLM:
  `VLLM_CACHE_ROOT=/snapshots`, SGLang:
  `TORCHINDUCTOR_CACHE_DIR=/snapshots/torch_inductor`).
- `3303c6f feat(snapshots):` background GC loop +
  `~/.serve/snapshots.yaml`. Re-reads config each tick (24h) so the
  operator doesn't need a daemon restart to flip knobs.

**Live-verified on RTX PRO 6000 Blackwell, qwen3-0_6b cold→warm:**

|        | wall clock | vLLM `torch.compile took` |
|--------|-----------:|---------------------------:|
| cold   | 128 s      | ~25 s                     |
| warm   | 109 s      | 3.58 s                    |

The compile-step itself is **7× faster** on warm restore. The
remaining wall-clock floor is CUDA graph capture, which can't be
persisted across processes in vLLM 0.20.2. The design's 3-5×
end-to-end estimate was optimistic for this version; ~20 s shaved
per redeployment is what the infrastructure actually delivers today.
The persisted cache is portable across vLLM upgrades only when the
image tag matches (`engine_image` is in the snapshot key).

### Sub-project C — complete and live-verified

Five commits land the predictor:

- `7136d6a feat(predictor):` rule-based predictor — time-of-day,
  sequencing, key-affinity. Pure functions over `usage_events`;
  scores normalized so the combined `candidates()` dedupes by
  `(base, adapter)` keeping max score.
- `8d5ca1f feat(predictor):` tick loop + adapter pre-warming.
  `PredictorTask` runs in the daemon lifespan, 30 s default tick.
  v2.0 scope is intentionally narrow: only **adapter** candidates
  pre-warm, and only when a ready base deployment exists. Bare-base
  pre-warming needs plan reconstruction (operator chose gpu_ids /
  ctx / max_loras at `serve run` time); deferred.
- `0097239 feat(predictor):` `serve predict [--stats]` CLI + admin
  endpoints. `--replay` is a follow-up — it needs a usage_events
  export format + an LRU baseline emulator.
- `a5238ca feat(predictor):` `~/.serve/predictor.yaml` config loader.
  Partial overrides supported; missing file / malformed YAML /
  missing keys all silently fall back to defaults.
- `dcab4c5 feat(predictor):` `usage_aggregates` rollup table (migration
  008) + daily `UsageRollupTask`. Events older than
  `retention_days` get aggregated into
  `(base, adapter, hour_of_week)` buckets; raw rows go away. Keeps
  the predictor's hot table bounded on long-running boxes.

**Live-verified end-to-end on RTX PRO 6000:**

1. Brought up qwen3-0_6b with `--max-loras 4 -x '--max-lora-rank=64'`.
2. Pulled `codelion/Qwen3-0.6B-accuracy-recovery-lora` as `test-lora`.
3. Seeded 5 historical `(qwen3-0_6b → test-lora)` sequencing pairs.
4. Fired one bare-base request — the sequencing trigger.
5. `serve predict` showed `qwen3-0_6b:test-lora` at score 0.417,
   reason `sequencing (P=0.42 after 'qwen3-0_6b' within 30s)`.
6. Within one tick the daemon logged
   `Loaded new LoRA adapter: name 'test-lora'`;
   `serve predict --stats` showed
   `preloads_attempted=1, preloads_succeeded=1`, then
   `preloads_skipped_already_warm=1` on the next tick.
7. Subsequent adapter request returned in **51 ms TTFB** — vs ~38 s
   without the predictor.
8. `usage_events.cold_loaded=0` for that request, confirming the
   Sub-project C success metric correctly reflects the win.
