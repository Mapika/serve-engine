### What I did NOT live-verify

I did not pull a real adapter from HuggingFace and exercise the
end-to-end flow on the real RTX PRO 6000 Blackwell. That's
deliberate — overnight without your eyes on it, an unknown HF repo
could:
- Be unavailable or permission-gated.
- Have an incompatible architecture for the registered base (LoRA
  rank mismatch, target-modules mismatch).
- Trigger a 30-60s engine restart that masks bugs.

The integration test exercises every code path with a fake engine
that records every load/unload/dispatch call, including the LRU
eviction sequence at slot capacity. That + the 245 passing unit tests
give high confidence the surface is correct; live verification is
~5 minutes of your time tomorrow.

### Live verification in 5 minutes

```bash
# 1. Bring daemon up on the new code
serve daemon stop && serve daemon start

# 2. Already-registered base from prior sessions (qwen3-0_6b)
#    is still here; make sure deployment has LoRA enabled
serve stop
serve run qwen3-0_6b --gpu 0 --engine vllm --ctx 2048 --max-loras 4

# 3. Pull a real public Qwen3-compatible LoRA
#    (suggested: search HF for 'qwen3 lora' for something small)
serve adapter pull <some-repo> --base qwen3-0_6b --name test-lora

# 4. Hit the proxy with the adapter as the model
curl -sN -X POST http://127.0.0.1:11500/v1/chat/completions \
  -H "Authorization: Bearer <your-key>" \
  -H 'Content-Type: application/json' \
  -d '{"model":"test-lora","messages":[{"role":"user","content":"hi"}],"max_tokens":40}'

# 5. Time a second request — should be sub-second to first byte
#    (no hot-load needed)

# 6. Cleanup
serve adapter rm test-lora --force
```

If you see anything weird, the daemon log at `~/.serve/logs/daemon.log`
will have the engine HTTP load/unload calls; the engine container's
`docker logs serve-vllm-qwen3-0_6b-<id>` will show the LoRA load
attempts.
